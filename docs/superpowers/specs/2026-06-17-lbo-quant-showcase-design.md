# LBO Quant Showcase Rebuild — Design

**Date:** 2026-06-17
**Branch:** `feat/quant-showcase-nextjs`
**Status:** Design — pending implementation

## Problem

The deployed showcase (`web/index.html`, generated by `tools/export_site.py`) is
static Jinja2 + embedded Vega-Lite. It surfaces only a sliver of what the engine
computes — an IRR leaderboard and one sensitivity heatmap — and reads as
"basic" next to the user's other Vercel sites (`capital-markets-intelligence`,
`sellside-research-engine`).

Two distinct gaps:

1. **Depth wasted.** `src/lbo_model.py::run_lbo` already produces a full,
   balancing three-statement model, a per-tranche debt waterfall, sources &
   uses, and exit re-rating. The site throws ~80% of it away.
2. **Presentation layer is the wrong tool.** Jinja-rendered static HTML cannot
   deliver the interactive, animated, premium feel the reference sites have.
   That feel comes from a React frontend with a real charting library — not from
   "deploying Python to Vercel" (which would mean serverless functions, the
   wrong tool for a dashboard).

This project rebuilds the **presentation layer** as a Next.js app, adds a new
**advanced-quant analytics layer** in Python, and **automates a weekly refresh**
— without modifying any existing `src/` model math.

## Goal

A premium, interactive, statically-served Next.js dashboard that (a) surfaces
the full depth the engine already computes, (b) adds a differentiated
advanced-quant layer (solvers, global sensitivity, downside risk, India
take-private feasibility), and (c) refreshes itself weekly via CI. Python is the
**engine**, not the website; it runs before Vercel and leaves a JSON artifact
behind.

## Scope (decided)

Product shape, depth modules, and quant modules were chosen interactively with
the user (visual brainstorming companion). All were selected — the full set.

**Product shape — Hybrid.** A dashboard landing (`/`) that drills into a
per-company tear sheet (`/t/[ticker]`). Maps onto the existing index + `/t/`
structure.

**Depth modules (engine already computes; surface them):**
- Per-tranche debt-paydown waterfall (from the `schedule` dataframe)
- Linked three-statement model (IS / cash flow / balance sheet), with
  common-size toggle
- Sources & Uses waterfall
- Premium × exit-multiple sensitivity heatmap + tornado (`sensitivity_grid_premium`)

**Advanced-quant modules (new `src/analytics.py`; each calls `run_lbo`):**
- `monte_carlo()` — distributions over revenue growth, margin, exit multiple →
  IRR/MOIC distribution, fan chart, P(beat hurdle)
- `irr_bridge()` — attribution of return to deleveraging / EBITDA growth /
  multiple re-rating / FCF
- `value_bridge()` — absolute (₹) equity-value-creation decomposition
- `max_bid_solver()` — bisection on entry premium for a target IRR hurdle
- `debt_capacity_solver()` — bisection on leverage s.t. min DSCR/ICR ≥ covenant
  every year
- `optimal_exit()` — IRR for exit at each year 1…N → IRR-maximizing hold
- `sobol_indices()` — global (variance-decomposition) sensitivity of IRR to
  drivers, including interactions
- `downside_risk()` — P(loss / capital impairment), 5% VaR, CVaR (expected
  shortfall) on MOIC, from the Monte Carlo sample
- `iso_irr_frontier()` — contour of (entry premium × exit multiple) combos that
  hit the target IRR exactly
- `feasibility_score()` — composite India take-private feasibility from promoter
  holding, pledge, free float, valuation gap
- `delisting_model()` — SEBI reverse-book-building: premium needed to cross the
  90% acceptance threshold, floor vs discovered price, and its IRR impact

**Automation.** Weekly GitHub Actions cron rebuilds the JSON and pushes; Vercel
auto-redeploys. Fails safe (keeps last good JSON on a bad fetch).

## Non-goals

- **No client-side LBO recomputation.** All math runs in Python at export time;
  the frontend reads precomputed numbers. (Interactive "what-if" sliders that
  recompute live are a future option, out of scope here — where a slider is
  wanted, Python precomputes a grid the UI indexes into.)
- **No modification of existing model math:** `src/lbo_model.py`,
  `src/screener.py`, `src/statements.py`, `src/data_loader.py`, `src/app.py`
  are untouched. New analytics live in a new module that imports them.
- **No Python runtime on Vercel.** No serverless functions; Vercel builds and
  serves static files only.
- **The Streamlit app stays** as the interactive companion; it is not ported.

## Architecture

```
config/config.yaml ─┐
data/*.csv ─────────┼─→ [existing src/ math]  ← UNTOUCHED
live yfinance ──────┘          │  run_lbo(), screener, statements
                               │
                               ▼
                     src/analytics.py  (NEW)
            solvers · Monte Carlo · Sobol · CVaR ·
            feasibility · delisting  — all call run_lbo()
                               │
                               ▼
                tools/export_data.py  (evolves export_site.py)
                               │
                               ▼
              web-app/public/data/results.json   ← the contract
                               │
                               ▼  (read at build time)
          web-app/  Next.js (static export) + Tailwind + ECharts
                /            → dashboard
                /t/[ticker]  → tear sheet
                               │
                               ▼
                vercel.json → Vercel builds & serves static output
                               ▲
        .github/workflows/weekly.yml (Mon cron) ─ runs export, commits
        results.json, pushes → Vercel redeploys
```

## Components

### 1. `src/analytics.py` (new)

The only new Python in `src/`. Pure, importable functions; no Streamlit import
(must be import-safe for the exporter and for tests). Each function takes the
same inputs `run_lbo` takes (entry revenue/EBITDA, assumptions, optional
overrides) and returns plain dicts/arrays.

- numpy for the stochastic and numerical-method pieces.
- A module-level default RNG seed so Monte Carlo / Sobol output is
  **reproducible** (parity tests depend on this).
- Solvers use bisection on the closed-form model (monotone in the swept
  variable over the relevant range); each returns the solved value **and** a
  convergence flag / residual so the UI can show "converged" vs "no solution in
  range" (e.g. a name that cannot clear the hurdle at any premium — handles the
  negative-EV case the JUSTDIAL fix already taught us about).

### 2. `tools/export_data.py` (evolves `tools/export_site.py`)

Orchestrator. Runs the screen, then for each passing name runs `run_lbo` + every
analytics function, and serializes everything to one `results.json`. Reuses the
existing `--no-fetch` / live-fetch paths from `export_site.py`. NaN/`inf` are
serialized as `null` (JSON-safe); the as-of date and config snapshot are stamped
in.

### 3. `web-app/` (new Next.js app)

- Next.js App Router, `output: 'export'` (pure static).
- Tailwind for the design system; a small set of tokens (color, type scale,
  spacing) for the premium look.
- ECharts (via `echarts-for-react`) as the charting workhorse — heatmaps, fan
  charts, gauges, bridges, contours, financial charts. A thin wrapper component
  enforces consistent theme/typography across all charts.
- Routes: `/` (dashboard) and `/t/[ticker]` (`generateStaticParams` from the
  passer list in `results.json`).
- Reusable widget components, one per chart type, each consuming a slice of the
  JSON.

### 4. `.github/workflows/weekly.yml` (new)

- `schedule: cron: '0 6 * * 1'` (Mondays) + `workflow_dispatch` (manual).
- Steps: checkout → setup Python → `pip install -r requirements.txt` →
  `python tools/export_data.py` → if the JSON is valid and non-empty, commit &
  push `results.json`; else fail the job and leave the last good JSON in place.
- Vercel's Git integration redeploys on the push. (No Vercel build hook needed.)

## Data contract — `results.json`

A single document is the interface between Python and the UI. Shape (illustrative):

```jsonc
{
  "as_of": "2026-06-17",
  "config": { "hurdle_irr": 0.20, "hold_years": 5, /* … snapshot */ },
  "universe": { "screened": 46, "passed": 5 },
  "passers": [
    { "ticker": "DMART.NS", "name": "Avenue Supermarts",
      "irr": 0.27, "moic": 2.4, "feasibility": 72, "max_bid_premium": 0.31 }
    /* … leaderboard-level summary, one row per passer */
  ],
  "companies": {
    "DMART.NS": {
      "statements": { "income": [...], "cash_flow": [...], "balance_sheet": [...] },
      "debt_schedule": [...],
      "sources_uses": {...},
      "returns": { "irr": 0.27, "moic": 2.4, "irr_bridge": {...}, "value_bridge": {...} },
      "montecarlo": { "irr": [...], "moic": [...], "p_beat_hurdle": 0.68 },
      "downside": { "p_loss": 0.04, "var5": 1.1, "cvar5": 0.9 },
      "sensitivity": { "grid": [...], "tornado": [...], "iso_frontier": [...] },
      "solvers": { "max_bid": {...}, "debt_capacity": {...}, "optimal_exit": {...} },
      "sobol": { "first_order": {...}, "total_order": {...} },

```

The exact schema is finalized in P1; once frozen it is the boundary the frontend
codes against. Per-company blocks may be split into per-ticker files if the
single document gets large — the contract stays the same.

## Phasing

Each phase is independently shippable and reviewable.

- **P1 — Analytics + contract.** `src/analytics.py` + `tools/export_data.py` +
  frozen `results.json` schema. Fully testable in Python; no frontend yet.
- **P2 — Dashboard.** Next.js scaffold, design tokens, ECharts wrapper, the `/`
  dashboard (KPI band, IRR leaderboard, feasibility, sensitivity heatmap).
- **P3 — Tear sheet.** `/t/[ticker]` with three-statements, debt waterfall, IRR
  & value bridges, Monte Carlo fan, downside risk, solvers, Sobol, delisting.
- **P4 — Automation + polish.** `weekly.yml`, motion/transition pass, `vercel.json`
  repoint, README/run-docs update.

## Testing

- **Python (per analytics fn):**
  - `max_bid_solver` result, re-run through `run_lbo`, lands on the hurdle within
    tolerance; returns "no solution" when the name can't clear it at any premium.
  - `debt_capacity_solver` result keeps min coverage ≥ covenant and is the
    binding maximum (one notch higher breaches).
  - `monte_carlo` / `sobol` reproducible under the fixed seed.
  - `downside_risk`: CVaR ≤ VaR ≤ median; P(loss) in [0,1].
  - `optimal_exit` returns a year in 1…N.
- **Parity:** numbers in `results.json` match a direct `run_lbo` call (the JSON
  layer adds no math), mirroring the existing static-site parity tests.
- **JSON safety:** no `NaN`/`Infinity` tokens; every passer in `passers` has a
  matching `companies` block.
- **Frontend:** `next build` (static export) succeeds; smoke-render of the
  dashboard and one tear sheet; the build fails loudly if `results.json` is
  missing or malformed.

## Kept / replaced

- **Kept:** all `src/` math, the Streamlit app (`src/app.py`), existing tests,
  `data/`, `config/`.
- **Legacy:** the current `web/` (Jinja+Vega export) and `tools/export_site.py`
  remain in history but are superseded by `web-app/` + `tools/export_data.py`.
  `export_data.py` may reuse `export_site.py` helpers it needs.
- **Changed:** `vercel.json` repoints to build/serve the Next.js app.

## Risks / open questions

- **yfinance reliability** in CI — mitigated by the fail-safe (keep last good
  JSON). Worth confirming the live-fetch path is deterministic enough for a
  scheduled job, or whether the committed `data/market_snapshot.csv` should be
  the CI default with live fetch as opt-in.
- **`delisting_model` assumptions** — reverse-book-building needs a tendering
  assumption set; the model must state its assumptions on the tear sheet (it's a
  scenario tool, not a prediction). Lowest-confidence module; could ship in P3 as
  "indicative."
- **Sobol cost** — needs many `run_lbo` evaluations per name; fine given the
  closed-form model, but worth a sample-size cap so the weekly job stays fast.
- **Single vs per-ticker JSON** — decide in P1 based on observed size.
